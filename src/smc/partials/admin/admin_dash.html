Dashboard

This is our web app backoffice. You should / will be able to execute certain project operations from here. Such as editing a view (partial file), or adding a new section to the site (and his link in the footer or navbar for example). Editing styles (css, with realtime preview).
And first for all, deploying automatically to production hosting with one click.

All the operations here should target firebase realtime database for smc. An instance of the web app project running in heroku will then listen all the changes and be in charge of compilation and publishing.

Views (partials)

Partial files are reusable pieces of code. The most common example is the follow: Your site has two pages: contact-page and about-us. These two pages has the same header and footer. So we abstract each section and create two partials. After, each pages only needs to include header and footer partial rather than duplicate the code.
To know more about partials, take a look at the handlebars framework. We use handlebars. Views are in the end a pure DRY principle. Do not repeat yourself.

Pages (static)

Following the example above, contact-page and about-us are the pages (static content). Each page file represents a section that can be acceded by the browser. The url depends exclusively of the location of the file in the project folder three (static folder).
Take a look at the address bar right now. /admin/index.html is the rendered result of the login page (named index.admin.html), meanwhile, /admin/dashboard/index.html is the result of an static pages named index.dash.html located in src/smc/static/admin/dashboard (project tree).
Static pages are a bunch of index.html files in the end, so that helps a lot keeping clean urls, following standards and improving SEO.

Pure HTML, CSS, JS

Web app project output (generated files) are in the end just client-side files. A bunch of html, css, js which follow web standards and they do not require any other technology aside the browser. No php, no mysql, no ruby, no node. We are able to publish these files to any normal hosting.

Packaging

We package content using different techniques. CSS and JS files are concatened in one file to improve distribution. JS also is packaged in different bundles defined by us in the partial files. JS also is compiled and minified using Babel (https://babeljs.io/), which ensure compatiliby for ES5 (legacy browser) while we keep coding using latest technology and ES7. We also generate HTML5 shim files https://github.com/es-shims/es5-shim to ensure old browser compatibility. Packaging ensures that we can write using preprocessors languages and new technologies and in the end, he is in charge of speeding things up and compatiliby concerns.

Publishing

The project already has ftp publishing in one command. You only need to specify the ftp credentials and ftp password. Ftp password is stored in enviromental variables.
The expectation is that we are going to be able to publish from here, in the backoffice, using a connected instance of the project running in heroku.
